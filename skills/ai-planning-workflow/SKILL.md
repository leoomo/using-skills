# AI Planning Workflow

基于 Boris Tane 的 AI 编程工作流，永远不要让 AI 写代码，直到审核批准书面计划。

## 用法

```bash
# 交互模式（推荐）- 输入任务描述
/ai-planning-workflow <任务描述>

# 或指定功能名称
/ai-planning-workflow <功能名称> <任务描述>
```

### 参数

| 参数 | 说明 |
|------|------|
| `<功能名称>` | 可选，功能目录名称 |
| `<任务描述>` | 必需，描述你想要实现的功能或任务 |

## 核心理念

**核心原则**: 永远不要让 AI 写代码，直到审核批准书面计划

在写任何代码之前，必须完成以下步骤：
1. Research - 深读相关代码，理解现状
2. Plan - 生成计划，包含关键决策点
3. 用户确认关键决策
4. **最终计划审阅** - 呈现完整计划供用户审核
5. Todo - 生成任务清单
6. Implement - 执行计划
7. **总结与记忆确认** - 总结经验，确认是否写入记忆

## 文件结构

```
当前项目目录/
└── ai-planning/
    └── <功能名称>/
        ├── research.md    # 研究结果
        ├── plan.md        # 实施计划
        └── todo.md        # 任务清单
```

## 工作流程

### Step 1: 启动与功能名称确认

1. 解析用户输入：
   - 如果只提供任务描述 → 询问功能名称
   - 如果提供 `<功能名> <任务>` → 使用功能名

2. 确认功能名称，获取用户认可

### Step 2: 创建目录结构

自动创建：
```
ai-planning/<功能名称>/
├── research.md
├── plan.md
└── todo.md
```

### Step 3: Research

1. 使用 Agent 工具的 Explore agent 深读相关代码
2. 搜索关键词、相关文件、理解现有架构
3. 将研究结果保存到 `research.md`

**Research 内容应包括**:
- 现有代码结构
- 相关模块和依赖
- 可能的实现方式
- 潜在的风险和挑战

### Step 4: Plan

基于 research 结果，生成实施计划：

**Plan 内容应包括**:
1. **目标**: 简述要实现的功能
2. **关键决策点**: 需要用户确认的技术选择
3. **实现步骤**: 详细的实现计划
4. **风险评估**: 可能的问题和应对措施

**关键决策点格式**:
```markdown
## 关键决策点

### 决策 1: <标题>
- 选项 A: <描述>
- 选项 B: <描述>
- 建议: <建议>
- 需要确认: (y/n)

### 决策 2: <标题>
...
```

### Step 5: 用户确认关键决策（优化版）

**新流程**:
1. **一次性展示所有决策点** - 用户可以一次性看到所有需要确认的决策
2. **批量确认** - 用户可以回复：
   - `y all` - 全部确认
   - `y 1,3` - 确认第1和第3项
   - `n 2` - 拒绝第2项（需要说明理由）
3. **逐个确认** - 如果用户选择逐个确认，则逐个展示

**展示格式**:
```
## 待确认决策 (共 X 项)

### 决策 1: <标题>
- 选项 A: <描述>
- 选项 B: <描述>
- 建议: <建议>

### 决策 2: <标题>
...

---
请确认 (可输入: y all / y 1,3,n 2 / 逐个确认):
```

### Step 6: 更新 Plan

根据用户确认，更新 plan.md，标注已确认的决策

### Step 7: 最终计划审阅（新增）

**这是关键步骤** - 决策完成后，必须将完整计划呈现给用户审阅：

```
═══════════════════════════════════════
       📋 最终实施计划 - <功能名称>
═══════════════════════════════════════

## 🎯 目标
<描述实现的功能/优化>

## ✅ 已确认的决策
1. 决策 1: <选择>
2. 决策 2: <选择>
...

## 📝 实现步骤

### Phase 1: <名称>
- [ ] 任务 1.1
- [ ] 任务 1.2

### Phase 2: <名称>
- [ ] 任务 2.1
- [ ] 任务 2.2

## ⚠️ 风险评估
- 风险 1: <描述> → <应对措施>
- 风险 2: <描述> → <应对措施>

## 📁 文件位置
- 研究: `ai-planning/<功能名称>/research.md`
- 计划: `ai-planning/<功能名称>/plan.md`
- 任务: `ai-planning/<功能名称>/todo.md`

═══════════════════════════════════════

请确认以上计划，回复:
- `确认` 或 `y` - 开始执行
- `修改` 或 `n` - 需要修改某个决策
- `取消` - 放弃此计划
═══════════════════════════════════════
```

### Step 8: Todo

基于确认后的 plan，生成详细的任务清单

**Todo 格式**:
```markdown
# Todo: <功能名称>

## 任务列表

- [ ] Task 1: 描述
- [ ] Task 2: 描述

## 执行顺序

1. Task 1
2. Task 2
```

### Step 9: Implement

执行 todo 列表中的任务，**每个任务带有自动验证**：

**Todo 格式增强**:
```markdown
- [ ] Task 1: 描述
  - 验证命令: npm run typecheck
  - 最大重试: 2

- [ ] Task 2: 描述
  - 验证命令: npm test
  - 最大重试: 3
```

**执行流程**:
```
For each task:
  1. 执行任务 (修改代码)
  2. 如果有验证命令:
     3. 运行验证命令
     4. ✅ 验证通过 → 标记完成，继续下一任务
     5. ❌ 验证失败:
        - 重试次数 < 限制 → 尝试修复 → 回到步骤 3
        - 重试次数 >= 限制 → 停止，报告错误
  6. 更新 todo.md 标记
  7. 继续下一任务
```

**验证命令类型**:
| 类型 | 示例 |
|------|------|
| npm 脚本 | `npm run typecheck`, `npm test` |
| Shell 命令 | `eslint src/`, `tsc --noEmit` |
| 无验证 | (任务不带验证命令则跳过) |

**风险与应对**:
| 风险 | 应对 |
|------|------|
| 验证命令不存在 | 跳过验证，显示警告 |
| 验证超时 | 设置超时限制 (60s) |
| 修复后仍失败 | 超过重试次数后停止 |

### Step 10: 总结与记忆确认

任务完成后，进行以下总结：

**总结内容**:
1. 完成情况回顾
2. 遇到的挑战与解决方案
3. 关键经验教训
4. 可复用的模式

**记忆写入指引**:
指导何时写入记忆：
- 发现新的可复用模式
- 遇到反复出现的问题
- 技术决策的重要理由
- 项目特定的约定或偏好

**总结输出格式**:
```
═══════════════════════════════════════
       📊 任务总结 - <功能名称>
═══════════════════════════════════════

## ✅ 完成情况
- 完成的任务: X/Y
- 总耗时: 约 X 分钟

## 💡 经验教训
1. <经验1>
2. <经验2>

## 🔄 可复用模式
- <模式描述>

## 📝 下次改进
- <改进点>

═══════════════════════════════════════

是否需要将以上经验写入记忆？
- `y` - 写入记忆
- `n` - 不写入
═══════════════════════════════════════
```

**记忆确认机制**:
- 默认不写入
- 仅当用户明确输入 `y` 时才写入
- 回车或输入其他内容默认不写入
- 项目记忆存放在当前项目的 `memory/` 目录

## 示例

```bash
# 交互模式 - 输入任务描述
/ai-planning-workflow 添加用户认证功能

# 指定功能名称
/ai-planning-workflow user-auth 实现登录注册功能
```

## 规则

- **永远不要跳过 Research**: 不了解现有代码就不能写计划
- **计划必须经过确认**: 关键决策必须用户点头才能继续
- **必须最终审阅**: 决策确认后必须展示完整计划供用户审核
- **先计划后代码**: 写代码前必须有已批准的 plan.md
- **持续验证**: 实现过程中持续 typecheck，发现问题立即修复

## 执行指令

当用户调用此技能时：

### 解析输入

1. 解析参数：`<功能名称>` 和 `<任务描述>`
2. 如果只有任务描述 → 询问功能名称

### 创建目录

```bash
mkdir -p ai-planning/<功能名称>
```

### 执行 Research

1. 启动 Explore agent 深读相关代码
2. 保存 research.md

### 生成 Plan

1. 基于 research 生成计划
2. 列出关键决策点
3. 保存 plan.md

### 用户确认（优化版）

1. **一次性展示所有决策点**
2. 提供批量确认选项 (y all)
3. 等待用户输入确认方式
4. 处理确认结果

### 最终计划审阅（新增）

1. 读取 plan.md 内容
2. 格式化呈现完整计划
3. 等待用户最终确认

### 生成 Todo

1. 基于确认的 plan 生成任务清单
2. 保存 todo.md

### 开始实现

1. 逐个执行任务
2. 运行验证命令
3. 验证失败 → 自动重试修复 → 重新验证
4. 验证通过 → 更新 todo.md 标记 → 继续下一任务
5. 所有任务完成 → 总结与记忆确认

### 总结与记忆确认

1. 生成任务总结：
   - 完成情况回顾
   - 遇到的挑战与解决方案
   - 关键经验教训
   - 可复用的模式

2. 展示总结输出（按格式）

3. 询问用户是否写入记忆：
   - 输入 `y` → 写入项目 `memory/` 目录
   - 回车或输入其他 → 不写入

4. 如果用户确认写入：
   - 创建 `memory/` 目录（如不存在）
   - 将经验写入相关记忆文件
