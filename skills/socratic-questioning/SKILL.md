# 苏格拉底式提问

当你需要进行深度分析、战略思考或多步推理时，请使用此提问框架。

## 快速开始

```bash
# 直接传入主题
/socratic-questioning 如何设计一个高可用的分布式系统

# 交互模式
/socratic-questioning
```

---

## 1. 三段式提问结构

将复杂问题拆解为三个递进层次：

### 第1部分：理论问题

```
什么使 [输出类型] 有效？
```

**目的**: 建立对问题本质的理解

**示例**:
- "什么使一份有效的产品需求文档？"
- "什么使一次成功的用户访谈？"
- "什么使一个可扩展的系统架构？"

### 第2部分：框架问题

```
什么原则或框架适用？
可能存在的反例或盲点是什么？
```

**目的**: 引入专业方法论，同时识别潜在风险

**示例**:
- "什么产品设计原则适用于这个场景？这些原则在什么情况下可能失效？"
- "什么数据分析框架能揭示问题根因？这个框架的盲点是什么？"
- "什么决策模型适合这种不确定性？什么证据会推翻这个决策？"

> **反向挑战机制**: 每个框架都需要回答"什么情况下这个方法不适用"，确保方案的健壮性。

### 第3部分：应用问题

```
现在将这些见解应用到 [你的具体任务]
```

**目的**: 将理论落地到具体情境

**示例**:
- "现在将这些原则应用到这个登录流程重设计"
- "现在用这个框架分析这份客户反馈数据"
- "现在基于这些见解制定技术选型方案"

## 2. 正确 vs 错误示例

| 错误（太宽泛） | 正确（苏格拉底式） |
|--------------|------------------|
| "分析这个客户反馈数据" | "客户反馈中的什么模式表明产品市场契合问题？什么定量和定性信号最重要？现在通过那个镜头分析这些数据。" |
| "写一个增长策略" | "什么渠道最适合这个产品的用户获取？什么指标能验证渠道有效性？现在基于这些标准设计增长策略。" |
| "优化这个代码" | "什么使这段代码难以维护？什么设计模式能改善结构？现在重构这个模块。" |

## 3. 问题堆叠技术

对于特别复杂的问题，可以堆叠多个问题：

```
[领域]专家在解决这类问题时会问什么？
他们需要什么数据？
他们会首先验证什么假设？

现在为我的[具体情况]回答这些问题，然后[执行任务]。
```

**完整示例**:

> "顶级增长营销人员在建立漏斗前会问什么？他们需要什么数据？他们会首先验证什么假设？
>
> 现在为我的SaaS回答这些问题，然后设计漏斗。"

## 4. 动态深度调整

根据任务复杂度选择提问深度：

### 完整版（三段式 + 反向挑战）

**适用**: 战略决策、架构设计、重大变更

```
1. 理论层：什么使 [输出] 有效？
2. 框架层：什么原则适用？反例/盲点是什么？
3. 应用层：如何应用到具体任务？
4. 质量检查：结果是否符合理论层的有效标准？
```

### 简易版（单层聚焦）

**适用**: 中等难度任务、快速迭代、时间敏感场景

```
1. 核心问题：解决这个问题的关键约束是什么？
2. 直接应用：基于这个约束，最佳方案是什么？
```

### 判断标准

| 复杂度 | 使用版本 | 特征 |
|--------|---------|------|
| 高 | 完整版 | 涉及多个利益方、长期影响、不可逆决策 |
| 中 | 简易版 | 单一目标、可迭代、影响范围可控 |
| 低 | 直接执行 | 常规操作、明确指令 |

## 5. 适用场景判断

### 适合使用（需要时）

| 场景 | 原因 |
|------|------|
| 战略思考 | 需要多维度分析 |
| 细微分析 | 需要深入理解本质 |
| 创意问题解决 | 需要打破思维定式 |
| 多步推理 | 需要分解复杂问题 |
| 决策制定 | 需要系统性评估 |

### 不适合使用（过度使用）

| 场景 | 原因 |
|------|------|
| 简单事实查询 | 直接回答更高效 |
| 数据格式化任务 | 不需要深度思考 |
| 基本代码生成 | 明确指令足够 |
| 快速改写 | 简单任务无需分解 |

## 6. 执行指令

当用户调用此技能时，按以下步骤执行：

### Step 1: 解析用户输入

如果用户提供了参数 `$ARGUMENTS`，将其作为待分析的主题/问题。
如果无参数，询问用户想分析什么主题或问题。

### Step 2: 识别问题复杂度

根据任务特征判断使用完整版还是简易版：

**快速判断**:
- 用户提到"战略"、"架构"、"设计"、"决策" → 完整版
- 用户提到"分析"、"优化"、"改进" → 简易版
- 用户提到"执行"、"实现"、"完成" → 直接执行

### Step 3: 应用提问结构

**完整版流程**:
1. **理论层**: 提问"什么使有效？" → 记录有效标准
2. **框架层**: 提问"什么原则适用？反例是什么？"
3. **应用层**: 提问"现在如何应用？"
4. **质量检查**: 基于理论层的有效标准，对结果进行自评

**简易版流程**:
1. **核心问题**: "关键约束是什么？"
2. **直接应用**: "最佳方案是什么？"

### Step 4: 必要时堆叠问题

对于复杂问题，先列出专家会问的问题，再要求具体执行

### Step 5: 输出结构化回答

确保回答体现：
- 对问题本质的理解
- 使用的方法论及其局限性
- 具体的应用方案
- **质量自评结果**（完整版必须）

## 7. 输出模板

### 完整版模板

```markdown
## 问题分析：[主题]

### 理论层：什么使 [目标输出] 有效？
- [ ] 要素 1
- [ ] 要素 2
- [ ] 要素 3

### 框架层：适用的原则/框架
- 原则 A: 说明

**潜在反例/盲点**：
- 什么情况下原则 A 可能失效？

### 应用层：针对 [具体任务] 的应用
1. 具体行动 1
2. 具体行动 2
3. 具体行动 3

### 质量自评
基于理论层的有效标准：
- [ ] 要素 1: ✅/❌ + 说明
- [ ] 要素 2: ✅/❌ + 说明
- [ ] 要素 3: ✅/❌ + 说明
```

### 简易版模板

```markdown
## 快速分析：[主题]

**关键约束**: [识别的核心限制]

**最佳方案**:
1. [直接行动]
```

---

**核心原则**: 你在编程 AI 的思维过程——通过精心设计的问题引导深度思考，而不是直接给出表面答案。
